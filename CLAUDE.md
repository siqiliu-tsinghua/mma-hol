# mma-hol ‚ÄĒ operational notes

Kernel-minimal higher-order logic theorem prover in Wolfram Language, LCF-style, modeled on HOL Light. Long-term target: mechanize undergraduate analysis through general Stokes and Fourier analysis, all within the Riemann-integral + Lebesgue-null-set framework (no full Lebesgue measure).

**Design document: `PLAN.md`** ‚ÄĒ authoritative for all architectural questions. This file is only the session-startup summary.

## Environment

- Wolfram Language 14.3 via `wolframscript` 1.13 (Linux). Notebooks are optional front-end only.
- All library code must be script-executable: `wolframscript foo.wls` ‚ÄĒ exit 0 pass, nonzero fail. This is a hard requirement, not a nice-to-have.

## Trust boundary (the one thing to never get wrong)

The **kernel** (types + terms + 10 primitive rules + axioms + `new_{constant,definition,basic_type_definition,axiom}`) is wrapped in a single `Module` whose `Unique["thm$"]`-gensymmed head is the sole legal constructor of `thm` values. Anything outside is untrusted. Worst-case bug in untrusted code is "can't prove a true theorem"; it must never be able to produce a false theorem.

Hard rules:

0. **Closure-based encapsulation is the project's originating design goal ‚ÄĒ non-negotiable.** All mutable kernel state (`thmTag`, `arityTable`, `constTypeTable`, `axiomList`, `defnList`, and any future registries) lives exclusively inside the `makeKernel[]` `Module` closure. External code ‚ÄĒ including other packages in the `HOL`` context tree ‚ÄĒ has no read or write path to these values; the only interface is the set of top-level functions the kernel installs. This models C++/Java-style private members with *true* inaccessibility via Wolfram's `Module` gensym mechanism. It overrides convenience: if a package elsewhere would be simpler with its own private `Association`, that's still not allowed. Previous code that kept state in a package-private context (e.g., M1's `arityTable` in `HOL`Types`Private``, M2's `constTypeTable` in `HOL`Terms`Private``) must be refactored to delegate into the kernel closure.
1. **`new_axiom` must disappear from the Kernel Association after bootstrap.** Three axioms (`ETA_AX`, `SELECT_AX`, `INFINITY_AX`) are declared during init; after that the API is withdrawn. Adding a new axiom is a user decision, not a Claude decision.
2. **Term construction above the kernel goes through `mkVar` / `mkConst` / `mkComb` / `mkAbs`.** Raw `comb[...]` / `abs[...]` literals bypass type-checking and őĪ-normalization ‚ÄĒ forbidden outside the kernel module.
3. **Kernel is a singleton.** `makeKernel[]` runs once at load; no reset. To clear state, restart `wolframscript`.
4. **Bound variables use a separate head `bvar[k_Integer, ty]`** produced by `mkAbs`. The scheme is de-Bruijn *levels-from-binder*: inside any `abs`, the immediately enclosing binder is `bvar[0, ty]`; references to a binder k levels further out are `bvar[k, ty]`. So `őĽx. őĽy. x` ‚Č° `abs[bvar[0, _], abs[bvar[0, _], bvar[1, _], "y"], "x"]` ‚ÄĒ each abs binds its own `bvar[0]`, and the `bvar[1]` in the innermost body points one level out. The original binder name lives in the 3rd slot of `abs[bv, body, origin]` for the printer only. Free vars (`var[name, ty]`) and bound vars (`bvar[k, ty]`) are *structurally* distinct heads, so no name-namespace conflict and no string parsing on the hot path. **őĪ-equivalence is `aconv`, not `===`** ‚ÄĒ `===` distinguishes the origin slot; `aconv` strips it. Kernel rules that compare terms up to őĪ must use `aconv`.
5. **Printer correctness is independent of soundness.** A printer bug that renders `‚äĘ False` as `‚äĘ True` won't break the kernel but will deceive readers ‚ÄĒ printer needs regression tests.

## Conventions

- **File extensions**: libraries in `.wl` (`BeginPackage`/`EndPackage`); entry points, tests, demos in `.wls`; interactive notebooks `.nb` are optional.
- **Contexts**: `HOL`Kernel``, `HOL`Bool``, `HOL`Real``, ‚Ä¶ Private helpers in `` `Private` `` subcontext.
- **Errors**: kernel throws `Throw[Failure["HOLError", <|"tag" -> ..., "msg" -> ...|>], holErrorTag]`; top-level `Catch` in wolframscript entries. Tests assert on tag. **Do not use `Message` + `$Failed`** ‚ÄĒ cannot distinguish failure modes.
- **Tests**: `tests/harness.wl` provides `assertEq` / `assertTrue` / `assertThrows` / `runTests` / `testExit`; `tests/run_all.wls` is the CI entry ‚ÄĒ it auto-discovers `tests/*_tests.wl`. Nonzero exit = failure.
- **Package loading**: `tests/run_all.wls` holds an ordered list of library files (`ErrorUtil.wl`, `tests/harness.wl`, `Types.wl`, ‚Ä¶) and `Get[]`s them in dependency order before discovering `tests/*_tests.wl`. Test files use `Needs["HOL`Whatever`"]` which is a no-op once the context is in `$Packages`. `Needs` cannot locate a flat-layout file on its own ‚ÄĒ always add new modules to the run_all load list.
- **Failure payload access**: `Failure[tag, assoc]` ‚ÄĒ `[[1]]` is the tag string, `[[2]]` is the association. Always read keys via `f[[2, "tag"]]`, never `f[[1, "tag"]]`.
- **No comments describing what code does** ‚ÄĒ names should carry that. Comments only for non-obvious *why*.
- **No `_` in identifiers**: WL parses `FOO_BAR` as `Pattern[FOO, Blank[BAR]]`, which silently introduces `FOO` and `BAR` as separate symbols in the current context ‚ÄĒ this can shadow imports from other packages. M5 originally used `CONJ_TAC`/`DISCH_TAC`/etc. and broke 77 tests by shadowing `HOL`Bool`CONJ`/`HOL`Kernel`ASSUME`/etc. The convention is camelCase for tactics (`conjTac`, `dischTac`, `popAssum`) and ALL-CAPS for combinators that don't contain underscores (`THEN`, `THENL`, `ORELSE`, `REPEAT`, `TRY`).
- **No `Function[{_}, body]`**: `_` (Blank) is not a valid Function parameter name; WL emits `Function::flpar` and leaves the call unevaluated, which then propagates as garbage values into kernel functions and trips a "not a theorem" error far from the cause. Use a real symbol, e.g. `Function[{ignored}, body]`.
- **Resource cost vs developer convenience**: if an implementation, algorithm, or trick exists to *save runtime, memory, or disk*, it is worth the implementation effort even when awkward. If a trick exists *only for developer convenience* (shorter code, fewer heads, easier patterns) but pays a runtime/memory/disk cost, prefer the costlier-to-write but cheaper-to-run alternative ‚ÄĒ especially when the code is on a hot path. Hot paths in this project are anything called once per `var`/`comb`/`abs` node during proof traversal: `freesIn`, `vsubst`, `instType`, `betaSubst`, `tyvarsInTerm`, `aconv`. Worked example: bound variables were originally `var["_b<k>", ty]` (one head, name-encoded index ‚Üí required `StringMatchQ` + `FromDigits` per node). Switched to a structurally separate `bvar[k_Integer, ty]` head ‚ÄĒ more pattern clauses across the codebase, but ~2√ó faster per traversal at typical term sizes (3.8√ó per check in isolation). The convenience price was real (every term-walking function gained a `bvar` clause); the runtime payoff scales linearly with proof size and is non-negotiable for M7+ scale.
